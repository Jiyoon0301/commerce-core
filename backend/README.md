# CommerceCore: 고성능 주문 및 재고 시스템

대규모 트래픽 환경에서도 안정적인 주문 처리를 목표로 설계한 분산 커머스 시스템입니다. 주문 생성과 같은 기능 그 자체보다, **Redis를 활용한 동시성 제어**와 **분산 환경에서의 데이터 정합성 보장**에 집중했습니다.

## 핵심 고민 및 해결 과정 (Troubleshooting)

### 1. 동시성 이슈: 200명이 동시에 주문하면 재고가 마이너스?

- **문제:** DB 락(Lock)을 사용해 재고를 차감하면 트래픽이 몰릴 때 응답 속도가 급격히 느려지는 병목 현상이 발생했습니다.
- **해결:** **Redis의 Lua 스크립트**를 도입했습니다. Redis는 단일 스레드로 동작하기 때문에, 재고 조회와 차감을 원자적(Atomic)으로 처리하여 락 없이도 안전하게 재고를 관리할 수 있었습니다. 이를 통해 동시성 테스트(200명 동시 주문)를 성공적으로 통과했습니다.

### 2. 네트워크 장애로 결제는 실패했는데, 재고는 이미 줄어들었다면?

- **문제:** 주문 생성, 재고 차감, 결제 요청이 각각 다른 시스템에서 일어날 때, 중간에 에러가 발생하면 재고 데이터가 꼬이는 문제가 있었습니다.
- **해결:** `OrderFacade` 패턴을 통해 **보상 트랜잭션(Compensating Transaction)**을 구현했습니다. 결제 실패 시 `try-catch` 블록 내에서 `restoreStock` 메서드를 호출하여, 차감된 재고를 강제로 복구함으로써 시스템 간 데이터 불일치를 방지했습니다.

### 3. 중복 클릭으로 같은 주문이 10번 들어오면?

- **문제:** 사용자가 결제 버튼을 빠르게 두 번 클릭하면 같은 주문이 중복 생성되는 현상이 발생했습니다.
- **해결:** `Idempotency Key(멱등성 키)`를 도입했습니다. Redis의 `SETNX(set if not exists)` 명령어를 활용하여, 주문 요청마다 고유한 키를 확인하게 함으로써 **동일한 주문 요청은 오직 한 번만 처리**되도록 방어 코드를 작성했습니다.

## 기술 스택

- **Backend:** Java 17, Spring Boot 3, JPA
- **Cache/Storage:** Redis (재고 관리, 멱등성 보장)
- **Testing:** JUnit 5, Mockito, Testcontainers (Redis 테스트 격리)

## 성과 및 학습

- **재고 복구 정합성 테스트 완료:** 테스트 코드 작성을 통해 재고 복구 로직의 실수를 찾아내고 수정하며, **비즈니스 로직과 데이터 정합성의 밀접한 관계**를 이해하게 되었습니다.
- **테스트 격리(Isolation):** 매 테스트마다 Redis의 키를 명확히 초기화하여, 테스트 간 데이터 간섭이 없는 환경을 구축했습니다.
